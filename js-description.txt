- какие идеи нужно реализовать;
- зачем нужна каждая часть логики;
- какие инструменты JavaScript для этого использовать;
- какие методы и функции подходят;
- что именно они значат и как помогают.

==================================================
1. С какой идеи нужно начать
==================================================

Когда разработчик делает калькулятор, первая ошибка обычно в том, что он сразу пытается "считать пример". Но калькулятор на самом деле состоит не только из вычисления.

Сначала нужно понять, что у приложения есть несколько задач одновременно:

1. Показать интерфейс и связать его с кодом.
2. Хранить текущее состояние.
3. Реагировать на действия пользователя.
4. Не давать пользователю вводить некорректные комбинации.
5. Уметь вычислять выражение.
6. Обновлять экран после каждого действия.
7. Поддерживать не только кнопки мышью, но и клавиатуру.
8. Показывать служебные статусы и подсказки.

То есть калькулятор нужно воспринимать как небольшое приложение со своим состоянием, событиями и правилами поведения.

Если смотреть на этот проект правильно, то JavaScript здесь решает три большие задачи:

- управляет состоянием;
- управляет вводом;
- управляет вычислением и отображением результата.

==================================================
2. Что нужно подготовить до написания основной логики
==================================================

Перед тем как писать вычисления, разработчику нужно получить доступ к элементам интерфейса.

Для этого используются методы DOM.

Главный метод в этом проекте:

`document.getElementById()`

Что он делает:
- ищет в HTML элемент по `id`;
- возвращает DOM-объект, с которым потом можно работать в JavaScript.

Пример идеи:

- получить экран текущего значения;
- получить экран предыдущего выражения;
- получить строку статуса;
- получить контейнер с кнопками.

Зачем это нужно:
- без этого код не сможет менять текст на экране;
- не сможет навесить обработчики событий;
- не сможет показывать результат вычислений.

В проекте для этого используются переменные:
- `currentEl`
- `previousEl`
- `statusEl`
- `controls`

Идея тут простая: сначала получить "точки контакта" между JavaScript и HTML, а уже потом строить логику.

==================================================
3. Почему обязательно нужно хранить состояние
==================================================

Следующая идея, которую нужно реализовать разработчику: калькулятор должен что-то помнить.

Если не хранить состояние в переменных, приложение не поймет:
- что сейчас введено;
- был ли уже получен результат;
- что показать в строке предыдущей операции.

Поэтому создаются переменные состояния:

- `expression` — текущее выражение;
- `previousExpression` — предыдущее выражение;
- `justCalculated` — признак того, что только что был показан результат.

Почему это важно:

Представим ситуацию:
- пользователь ввел `2+2`;
- нажал `=`;
- увидел `4`;
- потом нажал `7`.

Нужно решить: продолжать старое выражение или начать новое?

Именно для этого нужен `justCalculated`.

То есть состояние в калькуляторе нужно не "для удобства", а потому что без него невозможно грамотно описать поведение интерфейса.

==================================================
4. Почему выражение лучше хранить строкой
==================================================

Одна из ключевых идей: текущее выражение в калькуляторе лучше хранить строкой, а не числом.

Почему:

- пользователь вводит выражение постепенно;
- в процессе ввода строка может быть незавершенной;
- выражение может содержать операторы;
- ввод может выглядеть как `7+`, `10.`, `5*3`, `25%`.

Число не может хранить такие промежуточные состояния.

Строка может.

Поэтому выражение хранится как:

`let expression = '0';`

Это важное архитектурное решение. Оно означает:

- ввод сначала собирается как текст;
- потом при необходимости преобразуется и вычисляется.

Именно так работает большинство простых калькуляторов.

==================================================
5. Какие служебные идеи нужно предусмотреть заранее
==================================================

Кроме главного состояния, разработчику нужно подготовить несколько вспомогательных структур.

5.1. Набор допустимых клавиш

Для этого удобно использовать:

`new Set([...])`

Что такое `Set`:
- это коллекция уникальных значений;
- хорошо подходит для быстрых проверок "содержится элемент или нет".

Почему `Set` удобен здесь:
- нужно быстро понимать, разрешена ли нажатая клавиша;
- метод `has()` делает это удобно и понятно.

Пример идеи:

`validKeys.has(key)`

Что значит `has()`:
- проверяет, есть ли значение в наборе.

Зачем это нужно:
- калькулятор не должен реагировать на любые случайные клавиши;
- логика ввода должна быть ограничена только цифрами, точкой и операторами.

5.2. Отображение операторов

Внутри логики удобнее использовать:
- `*`
- `/`
- `-`

Но пользователю приятнее видеть:
- `×`
- `÷`
- `−`

Значит, разработчику нужно разделить:
- внутреннее представление;
- отображаемое представление.

Именно поэтому в приложении есть отдельные функции форматирования.

==================================================
6. Почему нужно сразу продумать обновление экрана
==================================================

Многие начинающие разработчики меняют DOM в разных местах напрямую. Это быстро приводит к хаосу.

Правильная идея:

- создать одну функцию, которая отвечает за обновление дисплея;
- после изменения состояния вызывать именно ее.

В проекте это делает функция:

`updateDisplay()`

Зачем нужна такая функция:
- логика интерфейса становится централизованной;
- легче поддерживать код;
- если меняется правило отображения, править нужно одно место.

Что она делает:
- берет текущее состояние;
- преобразует его к удобному для пользователя виду;
- записывает текст в DOM-элементы.

Для записи текста используется:

`element.textContent = ...`

Что такое `textContent`:
- свойство DOM-элемента;
- задает или читает текстовое содержимое элемента.

Почему именно `textContent`, а не `innerHTML`:
- здесь не нужно вставлять HTML-разметку;
- `textContent` безопаснее и проще;
- нет риска случайно вставить лишний HTML.

==================================================
7. Какие функции форматирования нужно придумать
==================================================

Когда разработчик строит калькулятор, важно отделить вычислительную форму от визуальной.

Для этого нужны как минимум две идеи:

1. Если значение пустое, на экране должен быть `0`.
2. Если выражение содержит технические символы, их нужно заменить на красивые.

Отсюда появляются вспомогательные функции:

- `formatDisplay(value)`
- `formatExpression(value)`

Какие методы строк здесь полезны:

`replace()`

Что делает `replace()`:
- заменяет часть строки на другую.

В проекте он нужен, чтобы:
- заменить `*` на `×`;
- заменить `/` на `÷`;
- заменить `-` на `−`.

Пример смысла:

`value.replace(/\*/g, '×')`

Что тут происходит:
- `/\*/g` — регулярное выражение для поиска всех звездочек;
- флаг `g` означает "заменить все совпадения";
- `'×'` — новое значение.

То есть разработчику нужно понимать: строковые методы в калькуляторе важны не меньше, чем математика.

==================================================
8. Почему нужны статусы и как их проектировать
==================================================

Если калькулятор только показывает числа, интерфейс получается "немым". Пользователь не понимает:

- выполнено действие или нет;
- произошла ошибка или нет;
- сейчас вводится выражение или уже зафиксирован результат.

Поэтому полезно реализовать строку состояния.

Для этого создается функция вроде:

`setStatus(text, type = 'normal')`

Здесь встречается важный прием:

`type = 'normal'`

Это значение по умолчанию для параметра функции.

Что это значит:
- если аргумент не передан, используется `'normal'`.

Почему это удобно:
- функцию можно вызывать короче;
- не нужно всегда явно писать тип сообщения.

Внутри такой функции можно менять:
- текст статуса;
- цвет статуса.

Для изменения стиля используется:

`element.style.color = ...`

Что это значит:
- JavaScript меняет inline-style конкретного элемента.

Этот подход подходит для небольшого динамического управления, когда состояние часто меняется.

==================================================
9. Какие базовые проверки должны быть в калькуляторе
==================================================

Перед тем как писать ввод, разработчику нужно определить правила:

- что считается оператором;
- как понять, какое число вводится сейчас;
- можно ли ставить точку;
- можно ли ставить два оператора подряд.

Для этого нужна отдельная вспомогательная логика.

9.1. Проверка операторов

Можно сделать функцию:

`isOperator(char)`

Она проверяет, принадлежит ли символ набору:
- `+`
- `-`
- `*`
- `/`
- `%`

Для этого подходит метод массива:

`includes()`

Что делает `includes()`:
- проверяет, содержит ли массив указанное значение.

Пример идеи:

`['+', '-', '*', '/', '%'].includes(char)`

Почему это удобно:
- запись короткая;
- логика хорошо читается.

9.2. Получение последнего числа в выражении

Если пользователь вводит точку, нужно понять: есть ли точка уже в текущем числе?

Для этого полезна функция вроде:

`getLastNumberChunk(str)`

Здесь удобно использовать:

`split()`

Что делает `split()`:
- разбивает строку на массив частей по разделителю.

Пример идеи:

`str.split(/[+\-*/%]/)`

Что это значит:
- строка делится по любому оператору;
- в итоге получается массив фрагментов;
- последняя часть массива — текущее вводимое число.

Дальше нужен доступ к последнему элементу:

`chunks[chunks.length - 1]`

Что значит `length`:
- длина массива.

Почему это нужно:
- только так можно корректно запретить вторую точку в одном числе.

==================================================
10. Как правильно проектировать ввод символов
==================================================

Главная идея калькулятора — функция добавления символа.

В этом проекте это:

`appendValue(value)`

Если разработчик делает калькулятор, именно эту функцию нужно продумать особенно внимательно. Она определяет почти весь пользовательский опыт.

Нужно решить несколько сценариев.

10.1. Если только что был результат

После вычисления пользователь может:
- либо продолжить выражение оператором;
- либо начать новое число.

Значит, нужна проверка:
- если `justCalculated === true`
- и новый символ не оператор,
- надо начать новый ввод.

10.2. Что делать с начальным нулем

Если текущее выражение равно `'0'`, а пользователь нажал `7`, экран должен стать `7`, а не `07`.

Значит, логика должна уметь заменять стартовый ноль.

10.3. Как вводить точку

Нужно обработать случаи:
- точка в обычном числе;
- точка после оператора;
- повторная точка в том же числе.

Для этого пригодятся:
- `slice()`
- `includes()`

`slice()`:
- возвращает часть строки.

Пример:

`expression.slice(-1)`

Что это делает:
- берет последний символ строки.

Это нужно, чтобы быстро понять, чем заканчивается текущее выражение.

`includes('.')`:
- проверяет, есть ли точка в текущем числе.

Если точка уже есть, нужно не молча ломать логику, а показать статус вроде "Уже есть точка".

10.4. Как вводить оператор

Если пользователь нажал оператор дважды подряд, обычно удобнее:
- не добавлять второй оператор рядом;
- заменить старый оператор новым.

Это делается так:
- проверить последний символ;
- если это оператор, заменить его.

Для замены удобно использовать:

`expression.slice(0, -1) + value`

Что это значит:
- взять всю строку без последнего символа;
- дописать новый оператор.

Идея очень важная: хороший калькулятор не просто принимает ввод, а корректирует типичные ошибки пользователя.

==================================================
11. Почему очистка и удаление должны быть отдельными действиями
==================================================

Разработчику нужно различать:

- полную очистку;
- удаление последнего символа.

Это разные сценарии, и смешивать их не стоит.

11.1. Полная очистка

Функция вроде `clearAll()` должна:
- вернуть начальное выражение;
- убрать прошлую операцию;
- сбросить флаг результата;
- обновить экран;
- показать успешный статус.

11.2. Удаление последнего символа

Функция вроде `deleteLast()` должна:
- удалить последний символ;
- если строка стала пустой, вернуть `0`.

Для удаления удобно использовать:

`slice(0, -1)`

Это один из самых полезных методов строк в подобных задачах.

Отдельный сценарий:
- если на экране уже зафиксирован результат, удаление может сбросить состояние целиком.

То есть разработчик должен заранее определить поведение после вычисления, а не оставлять его случайным.

==================================================
12. Как реализовать смену знака
==================================================

Операция `±` кажется простой только внешне.

На самом деле нужно решить два случая:

1. На экране одно число.
2. На экране выражение.

Если это просто число, можно менять знак напрямую.

Для проверки удобно использовать:

регулярное выражение и метод `test()`.

Пример идеи:

`/^-?\d*\.?\d+$/.test(expression)`

Что делает `test()`:
- проверяет, подходит ли строка под шаблон;
- возвращает `true` или `false`.

Что значит этот шаблон:
- число может начинаться с минуса;
- может содержать точку;
- должно оставаться корректным числом.

Если это число, знак можно поменять через:

`Number(expression) * -1`

Что делает `Number()`:
- преобразует строку в число.

Если это сложное выражение, сначала нужно его вычислить, а потом поменять знак результата.

Это хороший пример того, как одна пользовательская кнопка на самом деле требует разной внутренней логики.

==================================================
13. Как добавлять дополнительные операции
==================================================

Если разработчик хочет сделать калькулятор чуть функциональнее, полезно добавить:

- возведение в квадрат;
- квадратный корень.

Идея тут такая:

1. Взять текущее выражение.
2. Преобразовать его к вычисляемому виду.
3. Вычислить.
4. Применить нужную математическую операцию.
5. Сохранить прошлую операцию в понятном для пользователя виде.
6. Показать результат.

Для квадратного корня нужен метод:

`Math.sqrt()`

Что он делает:
- возвращает квадратный корень числа.

Но разработчику нужно помнить:
- корень из отрицательного числа в обычной арифметике показывать нельзя.

Значит, нужна явная проверка:

`if (value < 0) ...`

Это пример того, что дополнительные кнопки всегда требуют не только формулы, но и проверки граничных случаев.

==================================================
14. Почему отображение и вычисление нужно разделять
==================================================

На экране пользователь видит красивые символы:
- `×`
- `÷`
- `−`

Но JavaScript вычисляет:
- `*`
- `/`
- `-`

Следовательно, разработчику нужна функция подготовки выражения к вычислению.

В проекте это:

`sanitize(str)`

Ее смысл:
- взять визуальное выражение;
- привести к техническому виду.

Для этого снова используется `replace()`.

Это очень правильная идея проектирования:
- слой интерфейса не должен мешать слою вычисления;
- вычислитель должен получать понятные ему символы.

==================================================
15. Как вообще вычислять выражение
==================================================

Это центральный вопрос при разработке калькулятора.

Здесь важно понимать, что есть две отдельные задачи:

1. Проверить, что выражение безопасное и допустимое.
2. Вычислить его.

Нельзя просто брать любую строку и запускать ее как код.

Поэтому перед вычислением выражение валидируется.

Для проверки используется регулярное выражение:

`/^[0-9+\-*/%.\s]+$/`

Что оно означает:
- строка может содержать только цифры;
- операторы;
- точку;
- пробелы.

Здесь снова используется метод:

`test()`

Если строка не проходит проверку, нужно вызвать ошибку.

Для этого используется:

`throw new Error('invalid')`

Что делает `throw`:
- прерывает нормальное выполнение;
- выбрасывает исключение.

Что делает `new Error(...)`:
- создает объект ошибки.

Зачем это нужно:
- ошибки удобно перехватывать в `try...catch`;
- вычислительная логика становится управляемой.

==================================================
16. Как обрабатывать проценты
==================================================

Проценты в выражении нельзя напрямую вычислить как обычный оператор JavaScript в привычном смысле калькулятора.

Поэтому разработчику нужно заранее решить, как трактовать запись вроде:

`25%`

В проекте используется идея:

- заменить `25%` на `(25/100)`.

Для этого полезен `replace()` с регулярным выражением и группой захвата.

Пример идеи:

`str.replace(/(\d+(?:\.\d+)?)%/g, '($1/100)')`

Что тут означает `$1`:
- это содержимое первой захваченной группы.

Что означает `(?:...)`:
- это несохраняющая группа внутри регулярного выражения.

Разработчику не обязательно с первого раза идеально знать регулярные выражения, но важно понимать принцип:

- находим число с процентом;
- превращаем его в арифметическое деление на 100.

==================================================
17. Почему в проекте используется Function(...)
==================================================

После валидации выражение в проекте вычисляется через:

`Function('"use strict"; return (' + normalized + ')')()`

Что делает `Function(...)`:
- создает новую функцию из строки.

Почему после этого стоят `()`:
- первая пара создает функцию;
- вторая сразу ее вызывает.

То есть строка выражения превращается в вычисляемый код.

Важно понять смысл архитектурно:

- такой подход допустим только после строгой фильтрации символов;
- без проверки это было бы опасно.

Если бы разработчик делал более сложный или production-уровневый калькулятор, лучше было бы писать собственный парсер выражений. Но для учебного и небольшого проекта такой подход после валидации приемлем и достаточно компактен.

==================================================
18. Почему результат нужно форматировать отдельно
==================================================

Даже если выражение вычислено правильно, результат может выглядеть неаккуратно.

Типичная проблема JavaScript:

- дробные числа иногда отображаются с лишними хвостами;
- могут появляться значения вроде `0.3000000004`.

Поэтому нужна отдельная функция форматирования результата.

Например:

`formatResult(value)`

Что она должна проверить:

1. Число должно быть конечным.
2. Если число целое, не нужно оставлять десятичную часть.
3. Если дробное, его стоит округлить.

Здесь полезны методы и функции:

`Number.isFinite(value)`
- проверяет, является ли число конечным.

`Number.isInteger(value)`
- проверяет, является ли число целым.

`toFixed(10)`
- возвращает строку с числом, округленным до указанного количества знаков после запятой.

Дальше часто используется:

`Number(value.toFixed(10))`

Зачем снова `Number()`:
- чтобы убрать лишние нули в строке и вернуть нормальное числовое представление.

Это хороший пример того, что "получить ответ" и "красиво показать ответ" — не одно и то же.

==================================================
19. Как проектировать основную функцию вычисления
==================================================

Разработчику нужна функция верхнего уровня, например:

`calculate()`

Ее задача не в том, чтобы делать все самой, а в том, чтобы координировать процесс.

Правильная последовательность действий:

1. Подготовить выражение.
2. Вычислить его.
3. Отформатировать результат.
4. Сохранить прошлое выражение.
5. Обновить текущее состояние.
6. Показать успешный статус.
7. Пометить, что результат уже получен.

Это важный принцип проектирования:

- маленькие функции делают отдельные задачи;
- большая функция координирует сценарий пользователя.

==================================================
20. Почему ошибки нужно ловить через try...catch
==================================================

В калькуляторе легко возникают ошибки:

- выражение некорректно;
- попытка взять корень из отрицательного числа;
- результат бесконечен;
- вычисление не удалось.

Если не перехватывать ошибки, приложение будет ломаться или вести себя непредсказуемо.

Для этого используется конструкция:

`try { ... } catch { ... }`

Что значит `try`:
- попробовать выполнить код.

Что значит `catch`:
- если внутри `try` возникла ошибка, перехватить ее.

Почему это важно:
- пользователь получает понятное сообщение;
- интерфейс продолжает работать;
- код становится устойчивее.

Это один из главных приемов в любой интерактивной логике.

==================================================
21. Зачем нужны временные сообщения и таймеры
==================================================

Если показать сообщение об ошибке навсегда, интерфейс может стать неудобным. Иногда полезно показывать статус только ненадолго.

Для этого нужен таймер.

В проекте используется:

`setTimeout(...)`

Что делает `setTimeout`:
- запускает функцию один раз через указанное количество миллисекунд.

Также используется:

`clearTimeout(timer)`

Что делает `clearTimeout`:
- отменяет ранее запланированный таймер.

Зачем это нужно вместе:

- если пользователь быстро делает несколько действий подряд, старый таймер не должен перезаписать новый статус в неподходящий момент.

Это пример хорошего управления асинхронным поведением интерфейса.

==================================================
22. Почему кнопки действий нужно отделить от кнопок значений
==================================================

В интерфейсе калькулятора есть два разных типа кнопок:

- одни добавляют символ;
- другие запускают действие.

Например:
- `7` и `+` добавляют значение;
- `AC`, `⌫`, `=`, `√x` запускают команду.

Значит, разработчику нужно спроектировать две разные ветки логики.

В HTML для этого очень удобно использовать:

- `data-value`
- `data-action`

Что такое `data-*` атрибуты:
- это пользовательские атрибуты HTML;
- через них можно хранить данные прямо на элементе.

В JavaScript они читаются через:

`button.dataset.value`
`button.dataset.action`

Что такое `dataset`:
- объект со всеми `data-*` атрибутами элемента.

Это очень удобный и чистый способ связать кнопки с их назначением.

==================================================
23. Как правильно обрабатывать клики по кнопкам
==================================================

Вместо того чтобы навешивать `click` на каждую кнопку отдельно, удобнее использовать делегирование событий.

В проекте обработчик вешается на контейнер:

`controls.addEventListener('click', ...)`

Что делает `addEventListener()`:
- подписывает элемент на событие.

Почему это хороший подход:

- меньше кода;
- проще поддержка;
- если кнопки изменятся, контейнерная логика останется актуальной.

Внутри обработчика используется:

`event.target.closest('button')`

Что делает `closest()`:
- поднимается от исходного элемента вверх по DOM;
- ищет ближайший подходящий родительский элемент.

Почему это нужно:
- пользователь может кликнуть не строго по тексту, а по вложенной части кнопки;
- нужно надежно определить саму кнопку.

Также здесь используется:

`button.getBoundingClientRect()`

Что делает этот метод:
- возвращает размеры и координаты элемента на экране.

Зачем это нужно:
- вычислить место клика внутри кнопки;
- передать эти координаты в CSS для визуального эффекта.

И дальше:

`button.style.setProperty('--x', value)`

Что делает `setProperty()`:
- устанавливает CSS-свойство или CSS-переменную через JavaScript.

Это хороший пример связи JavaScript и CSS.

==================================================
24. Почему нужно отдельно поддержать pointerdown
==================================================

Кроме `click`, в проекте используется `pointerdown`.

Что это за событие:
- оно срабатывает в момент начала взаимодействия указателем;
- подходит для мыши, тача и пера.

Зачем это нужно:
- визуальный эффект нажатия должен появляться сразу, а не только после полного клика;
- интерфейс ощущается живее.

Это уже не про математику, а про качество пользовательского опыта.

==================================================
25. Как сделать визуальное нажатие кнопки из JavaScript
==================================================

Если пользователь работает клавиатурой, настоящего клика мышью нет. Но интерфейс все равно должен выглядеть отзывчивым.

Для этого полезно сделать функцию вроде:

`pressButtonVisual(selector)`

Что она делает:

1. Находит кнопку через `document.querySelector()`.
2. Добавляет класс `active`.
3. Через короткое время удаляет его.

Метод:

`querySelector()`

Что делает:
- ищет первый элемент, который подходит под CSS-селектор.

Методы:

`classList.add()`
`classList.remove()`

Что делают:
- добавляют и убирают CSS-класс.

Это позволяет запускать стили активного состояния не только мышью, но и программно.

==================================================
26. Как проектировать поддержку клавиатуры
==================================================

Хороший калькулятор должен работать не только мышью.

Для этого разработчику нужно подписаться на:

`document.addEventListener('keydown', ...)`

Событие `keydown`:
- срабатывает при нажатии клавиши.

Внутри обработчика используется:

`event.key`

Что это такое:
- строка с именем нажатой клавиши.

Зачем нужен `event.preventDefault()`:
- отменяет стандартное поведение браузера.

Это важно, например:
- чтобы Enter не вызывал нежелательные действия;
- чтобы Backspace не вел себя как переход назад в браузере в некоторых контекстах.

Дальше разработчику нужно разложить клавиши по сценариям:

- допустимые символы -> `appendValue()`;
- `Enter` и `=` -> `calculate()`;
- `Backspace` -> `deleteLast()`;
- `Shift + C` -> `clearAll()`.

Здесь важно понимать не только технику, но и идею:
- клавиатура должна использовать ту же бизнес-логику, что и кнопки;
- нельзя дублировать поведение разными независимыми кусками кода.

==================================================
27. Почему полезно сделать превью результата
==================================================

Если пользователь вводит выражение, удобно заранее показывать, чему оно примерно равно.

Это не обязательная функция, но она заметно улучшает интерфейс.

Для этого нужна идея "живого предварительного расчета":

- не вычислять слишком часто;
- не вычислять незавершенные выражения;
- не мешать основному результату.

В проекте это делает функция вроде:

`updateLivePreview()`

Она:
- ждет небольшую паузу;
- проверяет, можно ли вообще считать выражение;
- пробует вычислить;
- показывает превью в статусе.

Это пример дополнительной логики, которая не влияет на основную арифметику, но улучшает UX.

==================================================
28. Зачем здесь MutationObserver
==================================================

Интересная идея в проекте: предварительный расчет запускается не напрямую после каждого изменения переменной, а через наблюдение за изменениями дисплея.

Для этого используется:

`new MutationObserver(callback)`

Что делает `MutationObserver`:
- следит за изменениями DOM;
- вызывает функцию, когда эти изменения происходят.

Дальше используется:

`observer.observe(element, options)`

Что делает `observe()`:
- говорит наблюдателю, за каким элементом следить и какие типы изменений отслеживать.

Зачем это может быть полезно:
- логика превью привязывается к факту обновления интерфейса;
- меньше риска забыть вручную вызвать ее в каком-то месте.

То есть разработчик строит не просто набор функций, а связи между слоями интерфейса.

==================================================
29. Как проектировать код так, чтобы он оставался понятным
==================================================

Если посмотреть на готовую реализацию правильно, можно увидеть несколько хороших инженерных идей.

29.1. Разделение ответственности

Отдельно существуют:
- функции отображения;
- функции вычисления;
- функции изменения состояния;
- обработчики событий;
- вспомогательные проверки.

Это делает код легче для понимания и доработки.

29.2. Маленькие функции

Лучше иметь:
- `isOperator()`
- `sanitize()`
- `formatResult()`
- `clearAll()`

чем одну огромную функцию на все случаи жизни.

29.3. Общие точки входа

Для ввода есть `appendValue()`.
Для вычисления есть `calculate()`.
Для интерфейса есть `updateDisplay()`.

Это делает поведение предсказуемым.

29.4. Сначала состояние, потом DOM

Хорошая практика:
- сначала изменить переменные;
- потом отрисовать новое состояние.

Так код проще контролировать.

30. Какие методы и функции JavaScript особенно важны в этом проекте
==================================================

Ниже краткий список ключевых методов и их смысла.

`document.getElementById(id)`
- находит элемент по `id`.

`document.querySelector(selector)`
- находит первый элемент по CSS-селектору.

`document.querySelectorAll(selector)`
- находит все подходящие элементы.

`addEventListener(type, handler)`
- подписывает элемент на событие.

`event.preventDefault()`
- отменяет стандартное действие браузера.

- элемент, на котором произошло событие.

- ищет ближайший подходящий родительский элемент.

- возвращает координаты и размеры элемента.

`textContent`
- получает или задает текст внутри элемента.

`style.setProperty(name, value)`
- задает CSS-свойство или CSS-переменную.

`classList.add(name)`
- добавляет класс.

`classList.remove(name)`
- удаляет класс.

`replace(pattern, replacement)`
- заменяет часть строки.

`split(separator)`
- разбивает строку на массив частей.

`slice(start, end)`
- возвращает часть строки или массива.

`includes(value)`
- проверяет наличие значения.

`has(value)`
- проверяет наличие значения в `Set`.

`test(string)`
- проверяет строку по регулярному выражению.

`Number(value)`
- преобразует значение в число.

`Math.sqrt(value)`
- считает квадратный корень.

`Number.isFinite(value)`
- проверяет, конечно ли число.

`Number.isInteger(value)`
- проверяет, целое ли число.

`toFixed(n)`
- округляет число до `n` знаков после запятой и возвращает строку.

`setTimeout(fn, delay)`
- запускает функцию позже.

`clearTimeout(id)`
- отменяет запланированный таймер.

`Function(code)`
- создает функцию из строки.

`throw new Error(message)`
- выбрасывает ошибку.

`try...catch`
- позволяет безопасно обрабатывать ошибки.

`MutationObserver`
- отслеживает изменения DOM.

32. Итог
==================================================

Если объяснять создание JavaScript-логики этого калькулятора правильно, то суть не в том, чтобы "просто написать несколько функций".

Нужно последовательно реализовать идеи:

- хранить состояние калькулятора;
- отделить отображение от вычисления;
- валидировать ввод;
- централизованно обновлять интерфейс;
- поддержать разные сценарии поведения после результата;
- обрабатывать ошибки без поломки интерфейса;
- связать мышь, клавиатуру и визуальные эффекты в единую систему.

Именно это делает `script.js` полноценной логикой приложения, а не просто набором обработчиков.

JavaScript в этом проекте нужен для того, чтобы превратить статический HTML-интерфейс в работающий калькулятор, который:

- понимает действия пользователя;
- хранит и изменяет состояние;
- правильно собирает выражение;
- вычисляет результат;
- показывает статусы и подсказки;
- визуально реагирует на ввод.

То есть разработчику здесь важно думать не только о математике, но и о сценариях поведения, структуре кода, DOM-методах, строковых операциях, событиях и обработке ошибок. Именно из этого и собирается качественная JS-логика калькулятора.
